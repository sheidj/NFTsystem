# 高校毕业纪念NFT系统 - 答辩技术文档

> **文档说明**：本文档面向答辩评审，重点阐述系统的业务逻辑实现原理，全面覆盖答辩可能遇到的问题。

---

## 一、项目概述与核心问题

### 1.1 项目要解决的核心问题

**传统毕业纪念的三大痛点：**

1. **证书防伪困难**
   - 纸质证书易伪造、难验证
   - 学历查询依赖中心化数据库
   - 跨国验证流程繁琐

2. **纪念品缺乏价值**
   - 普通纪念品同质化严重
   - 无收藏价值，易丢弃
   - 无法代表个人独特性

3. **校友身份断裂**
   - 毕业后与学校失去联系
   - 校友身份难以证明
   - 校友网络难以维系

**区块链+NFT的解决方案：**

| 痛点 | 传统方式 | 区块链方案 | 优势 |
|------|---------|-----------|------|
| 证书防伪 | 纸质+中心化查询 | NFT上链 | 不可篡改、全球可验证 |
| 纪念品价值 | 批量生产 | 独一无二 | 稀缺性、可收藏、可转让 |
| 校友身份 | 校友卡易丢失 | NFT永久持有 | 数字身份、权益凭证 |

### 1.2 系统核心定位

**双轨制NFT体系：**
- **官方NFT（身份认证层）**：毕业证书、荣誉证书等，代表学校官方认证
- **个人NFT（纪念收藏层）**：盲盒铸造的纪念品，代表个人独特记忆

**关键设计原则：**
- 官方NFT不可转让（身份绑定）
- 个人NFT可转让（情感流转）
- 所有操作上链透明可查

## 二、核心功能逻辑详解

### 2.1 毕业生注册逻辑

**业务场景**：学校管理员将毕业生信息录入系统，建立区块链身份

**核心逻辑流程：**

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  输入信息    │ → │  校验唯一性  │ → │  写入区块链  │ → │  同步数据库  │
│ 地址/学号等  │    │ 学号/地址   │    │ 创建身份    │    │ 建立索引    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**关键逻辑点：**

1. **唯一性校验机制**
   - 学号唯一：防止重复注册同一学生
   - 地址唯一：确保一人一地址，防止身份冒用
   - 逻辑：查询链上数据，若已存在则拒绝

2. **权限控制逻辑**
   - 仅管理员角色可执行注册
   - 合约层使用OpenZeppelin的AccessControl
   - 前端也进行权限校验（双重保险）

3. **批量注册优化逻辑**
   - 问题：逐个注册10人需要10次交易，耗时久
   - 解决：循环处理，自动跳过已注册者
   - 进度反馈：实时显示处理进度（3/10）

**可能遇到的问题：**
- **Q**: 如果学号输错了怎么办？  
  **A**: 学号上链后不可修改，这是区块链的特性。建议管理员仔细核对，或注销后重新注册。

- **Q**: 一个学生可以用多个地址注册吗？  
  **A**: 不能。地址唯一性校验会拒绝。这样设计是为了确保"一人一身份"，防止刷NFT。

---

### 2.2 官方NFT发放逻辑

**业务场景**：学校向已注册毕业生发放官方认证NFT（毕业证书等）

**核心逻辑流程：**

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  选择NFT    │ → │  防重复检查  │ → │  链上铸造    │ → │  状态更新    │
│  类型/地址   │    │ 是否已拥有   │    │ 转移所有权   │    │ 标记已领取   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**关键逻辑点：**

1. **防重复发放机制（核心）**
   
   **为什么重要？** 官方NFT代表学校认证，重复发放会导致：
   - 同一学生拥有多个毕业证书（不合理）
   - 统计混乱（总量超过实际毕业生数）
   - 损害NFT稀缺性和价值

   **双层防护逻辑：**
   
   | 层级 | 位置 | 逻辑 | 作用 |
   |------|------|------|------|
   | 第一层 | 前端UI | 已拥有的NFT按钮禁用，显示"已发放" | 防止误操作 |
   | 第二层 | 智能合约 | `require(balanceOf(to, tokenId) == 0)` | 强制拦截 |

   **为什么需要两层？**
   - 前端层：提升用户体验，避免无效交易（省Gas）
   - 合约层：最终安全保障，即使绕过前端也无法重复发放

2. **四种NFT类型设计逻辑**

   | Token ID | 名称 | 供应量 | 设计意图 |
   |----------|------|--------|----------|
   | 1 | 毕业证书 | 10,000 | 全员基础认证 |
   | 2 | 纪念徽章 | 10,000 | 全员纪念 |
   | 3 | 荣誉证书 | 1,000 | 优秀毕业生（10%）|
   | 4 | 特殊奖项 | 100 | 杰出贡献者（1%）|

   **稀缺性梯度设计**：从全员拥有到极少数拥有，形成收藏层级

3. **批量发放逻辑**
   - 选择多个学生 → 选择NFT类型 → 一键发放
   - 自动跳过已拥有者（不报错，继续处理其他人）
   - 结果统计：成功X人 / 跳过Y人（已拥有）/ 失败Z人

**可能遇到的问题：**
- **Q**: 为什么官方NFT不能转让？  
  **A**: 官方NFT代表学校对毕业生身份的认证。如果可转让，A可以把"毕业证书"转给B，那B就成了冒牌毕业生。这是身份凭证，不是普通商品。

- **Q**: 如果发错了地址怎么办？  
  **A**: 无法撤回。区块链操作不可逆。所以发放前必须仔细核对地址，建议先小额测试。

### 2.3 个人NFT铸造逻辑（核心创新）

**业务场景**：已注册毕业生铸造独一无二的个人纪念NFT（盲盒机制）

**核心逻辑流程：**

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  点击铸造    │ → │  资格检查    │ → │  稀有度生成  │ → │  NFT铸造    │
│             │    │ 注册/次数   │    │ 随机+保底   │    │ 上链+IPFS   │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**关键逻辑点：**

1. **资格检查逻辑**
   - 必须已注册为毕业生（防止外部人员参与）
   - 每日限制5次（防止脚本刷取、控制通胀）
   - 每日次数北京时间0点重置

2. **盲盒稀有度生成逻辑（核心）**

   **设计目标：**
   - 公平：每个人都有机会获得稀有NFT
   - 有趣：类似抽卡游戏的刺激感
   - 防非酋：保底机制防止运气极差

   **三种稀有度及概率：**
   
   | 稀有度 | 概率 | 特点 |
   |--------|------|------|
   | 普通 | 70% | 基础款，颜色较简单 |
   | 稀有 | 25% | 进阶款，颜色更丰富 |
   | 传说 | 5% | 顶级款，带特效动画 |

   **保底机制逻辑：**
   
   ```
   每次铸造：pityCounter++
   
   if (pityCounter >= 20) {
       // 保底触发
       80%概率出稀有
       20%概率出传说
       pityCounter = 0  // 重置
   } else {
       // 正常概率
       按70%/25%/5%抽取
       if (出稀有或传说) pityCounter = 0
   }
   ```

   **保底机制的意义：**
   - 最坏情况：连续19次普通，第20次必出稀有/传说
   - 心理安慰：玩家知道不会永远非酋
   - 经济平衡：控制传说NFT总量（约5-6%）

3. **唯一性生成逻辑**

   **颜色种子生成：**
   - 基于：时间戳 + 区块随机数 + 用户地址 + TokenID + 总铸造数
   - 结果：256位整数（天文数字级别的唯一性）
   - 作用：确保每个NFT的视觉外观独一无二

   **铸造编号：**
   - 全局递增（第1个、第2个、第3个...）
   - 意义：收藏价值（编号越小越稀有）
   - 示例："毕业纪念 #88" 比 "#8888" 更有收藏意义

4. **每日限制逻辑**

   **为什么限制5次/天？**
   - 防止脚本刷取（控制通胀）
   - 增加稀缺性（不能无限铸造）
   - 鼓励日常参与（每天来铸造）

   **实现逻辑：**
   ```
   链上存储：lastMintDay[用户地址] = 上次铸造日期
   链上存储：dailyMintCount[用户地址] = 今日已铸造次数
   
   if (今天日期 != lastMintDay) {
       dailyMintCount = 0  // 新的一天，重置
   }
   require(dailyMintCount < 5)
   ```

**可能遇到的问题：**
- **Q**: 随机数可以被预测吗？  
  **A**: 使用block.timestamp + block.prevrandao + msg.sender + nonce的组合。在PoS机制下，prevrandao提供了链上随机性，矿工无法操控。对于盲盒场景足够安全。

- **Q**: 如果连续20次都是普通，保底会出传说吗？  
  **A**: 保底触发时，80%概率出稀有，20%概率出传说。所以保底不一定出传说，但一定不会是普通。

- **Q**: 为什么个人NFT可以转让？  
  **A**: 个人NFT是纪念品，不是身份凭证。A可以把纪念品送给B，就像送一个毕业礼物。官方NFT代表"你是毕业生"，个人NFT代表"你有一个纪念品"。

### 2.4 NFT转让逻辑

**业务场景**：用户将自己的个人NFT转让给其他地址

**核心逻辑流程：**

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  选择NFT    │ → │  输入地址    │ → │  执行转让    │ → │  所有权变更  │
│  个人/官方   │    │  接收方     │    │  链上转移    │    │  数据库同步  │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**关键逻辑点：**

1. **转让权限控制**
   - 仅个人NFT可转让（Token ID >= 10001）
   - 官方NFT禁止转让（Token ID 1-4）
   - 合约层强制执行，前端仅做提示

2. **安全校验逻辑**
   - 不能转给自己（无意义操作）
   - 地址格式校验（防止输错）
   - 余额校验（不能转让超过拥有的数量）

3. **所有权变更流程**
   - 链上：调用ERC-1155的safeTransferFrom
   - 链下：监听TransferSingle事件，更新数据库owner字段
   - 前端：刷新展示，显示新的拥有者

**可能遇到的问题：**
- **Q**: 转让后，铸造者信息还在吗？  
  **A**: 在。NFT元数据中永久记录了"铸造者"和"当前拥有者"。转让只改变所有权，不改变铸造历史。

- **Q**: 可以批量转让吗？  
  **A**: 当前版本仅支持单个转让。批量转让涉及复杂的权限和数量管理，作为毕业设计保持简洁。

### 2.5 数据存储与同步逻辑

**三层存储架构：**

```
┌─────────────────────────────────────────────────────────┐
│  第一层：区块链（链上）                                    │
│  - 存储：NFT所有权、毕业生信息、铸造记录                   │
│  - 特点：不可篡改、去中心化、查询慢                        │
└─────────────────────────────────────────────────────────┘
                            ↕ 事件监听 + API同步
┌─────────────────────────────────────────────────────────┐
│  第二层：PostgreSQL（链下）                                │
│  - 存储：用户索引、统计聚合、活动日志                      │
│  - 特点：查询快、支持复杂统计、便于分析                    │
└─────────────────────────────────────────────────────────┘
                            ↕ Pinata API
┌─────────────────────────────────────────────────────────┐
│  第三层：IPFS（去中心化存储）                              │
│  - 存储：NFT图片（SVG）、元数据（JSON）                    │
│  - 特点：永久存储、内容寻址、去中心化                      │
└─────────────────────────────────────────────────────────┘
```

**数据同步逻辑：**

1. **何时同步？**
   - 注册成功后 → 立即同步用户信息
   - 铸造成功后 → 立即同步NFT记录
   - 转让成功后 → 更新owner字段

2. **如何保证一致性？**
   - 使用区块链时间戳（block.timestamp）作为标准时间
   - 数据库记录包含链上交易哈希（可追溯）
   - 关键业务逻辑以链上数据为准，数据库仅用于展示

3. **为什么需要链下数据库？**
   - 区块链查询慢（需要RPC调用）
   - 不支持复杂聚合查询（如"按学院分组统计"）
   - 链上存储贵（需要Gas费）

**可能遇到的问题：**
- **Q**: 如果数据库丢了怎么办？  
  **A**: 可以从区块链重新同步。所有核心数据都在链上，数据库只是索引。重新扫描事件日志即可重建。

- **Q**: 数据库和链上数据不一致怎么办？  
  **A**: 以链上数据为准。系统设计时，关键操作（如是否已注册）都直接查询链上，不依赖数据库。

## 三、技术架构与选型逻辑

### 3.1 为什么选择ERC-1155而不是ERC-721？

**ERC-721（传统NFT标准）：**
- 一个合约只能管理一种NFT
- 批量操作需要多次交易（Gas高）
- 适合：CryptoPunks等单一类型收藏品

**ERC-1155（多代币标准）：**
- 一个合约管理多种NFT（官方4种 + 个人无限）
- 支持批量转账（节省Gas）
- 适合：游戏道具、本系统的多类型NFT

**本系统选择ERC-1155的原因：**
1. **成本**：部署一个合约 vs 部署多个合约
2. **效率**：批量发放NFT时，一次交易可发给多人
3. **扩展性**：未来可增加新的NFT类型，无需部署新合约

### 3.2 为什么使用PostgreSQL而不是纯链上存储？

**纯链上存储的问题：**
- 查询慢：每次查询需要RPC调用
- 费用高：存储数据需要Gas
- 功能弱：不支持SQL聚合查询

**PostgreSQL的价值：**
- 快速查询：毫秒级响应
- 复杂统计：GROUP BY、COUNT等聚合
- 数据分析：便于生成报表和图表

**设计原则：**
- 关键数据（所有权、身份）→ 链上
- 索引数据（统计、日志）→ 链下
- 媒体文件（图片）→ IPFS

### 3.3 为什么使用IPFS而不是中心化服务器？

**中心化服务器的问题：**
- 服务器宕机 → 图片无法显示
- 公司倒闭 → 数据丢失
- 篡改风险 → NFT图片被更换

**IPFS的优势：**
- 永久存储：内容寻址，只要有节点存储就不会丢失
- 去中心化：不依赖单一服务器
- 不可篡改：CID唯一标识内容，修改后CID变化

**实际应用：**
- NFT图片存储在IPFS，链上存储CID
- 即使项目方服务器关闭，NFT图片仍可访问
- 符合"去中心化"的区块链精神

## 三、智能合约设计

### 3.1 合约架构

**合约名称**：`GraduationNFT.sol`  
**标准**：ERC-1155（多代币标准）  
**部署网络**：Sepolia测试网

**核心数据结构：**

```solidity
// 毕业生信息
struct Graduate {
    string studentId;       // 学号（唯一标识）
    string studentName;     // 姓名
    string major;           // 专业
    string college;         // 学院
    uint256 mintedAt;       // 首次铸造时间
    bool hasClaimed;        // 是否已领取官方NFT
    uint256 selfMintCount;  // 个人NFT铸造次数
    uint256 pityCounter;    // 保底计数器
}

// 个人NFT数据
struct PersonalNFTData {
    address owner;          // 所有者
    Rarity rarity;          // 稀有度（0普通/1稀有/2传说）
    uint256 mintTime;       // 铸造时间
    uint256 colorSeed;      // 颜色种子（唯一）
    uint256 mintNumber;     // 铸造编号（全局递增）
}
```

### 3.2 核心功能实现

#### 3.2.1 毕业生注册

```solidity
function registerGraduate(
    address graduateAddress,
    string memory studentId,
    string memory studentName,
    string memory major,
    string memory college
) external onlyRole(ADMIN_ROLE)
```

**设计要点：**
- 仅管理员可注册（权限控制）
- 学号唯一性校验（防重复注册）
- 地址唯一性校验（一人一地址）

#### 3.2.2 官方NFT发放

```solidity
function mintOfficialNFT(address to, uint256 tokenId) 
    external onlyRole(MINTER_ROLE)
```

**防重复发放机制：**
```solidity
// 合约层强制检查
require(balanceOf(to, tokenId) == 0, "Already owns this NFT");
```

**四种官方NFT类型：**
| Token ID | 名称 | 供应量 | 稀有度 |
|---------|------|--------|--------|
| 1 | 毕业证书 | 10,000 | 普通 |
| 2 | 纪念徽章 | 10,000 | 普通 |
| 3 | 荣誉证书 | 1,000 | 稀有 |
| 4 | 特殊奖项 | 100 | 传说 |

#### 3.2.3 个人NFT铸造（核心创新点）

```solidity
function mintPersonalNFT() external returns (uint256)
```

**盲盒机制设计：**

```solidity
// 稀有度概率配置
uint256 constant LEGENDARY_RATE = 50;    // 5% 传说
uint256 constant RARE_RATE = 250;        // 25% 稀有（累计）
uint256 constant PITY_COUNT = 20;        // 20次保底

function mintPersonalNFT() external returns (uint256) {
    // 1. 验证资格
    require(已注册为毕业生);
    require(今日铸造次数 < 5);
    
    // 2. 保底机制
    graduates[msg.sender].pityCounter++;
    Rarity rarity;
    
    if (graduates[msg.sender].pityCounter >= PITY_COUNT) {
        // 保底触发：80%稀有 + 20%传说
        rarity = (random < 20) ? LEGENDARY : RARE;
        pityCounter = 0;
    } else {

        if (random < 50) rarity = LEGENDARY;
        else if (random < 300) rarity = RARE;
        else rarity = COMMON;

    uint256 colorSeed = keccak256(...);
    
    // 4. 铸造NFT

}
```

- 连续19次未出稀有/传说，第20次必出（80%稀有，20%传说）
- 出稀有/传说后，保底计数器重置
- 增加游戏性和公平性

## 附录：答辩常见问题深度解析（Q&A）

### Q1: 这个项目解决了什么实际问题？和传统电子证书有什么区别？

**解决的三大痛点：**

1. **证书防伪困难**
   - 传统：纸质证书易伪造，依赖中心化数据库验证
   - 本系统：区块链NFT不可篡改，全球公开可验证，无需信任第三方

2. **纪念品缺乏价值**
   - 传统：批量生产的纪念品同质化，无收藏价值
   - 本系统：每个NFT独一无二（颜色、编号、稀有度都不同），可收藏可转让

3. **校友身份断裂**
   - 传统：毕业后与学校失去联系，校友身份难以证明
   - 本系统：NFT作为永久数字身份，可随时验证校友身份

**与传统电子证书的本质区别：**

| 维度 | 传统电子证书 | 区块链NFT |
|------|-------------|----------|
| 所有权归属 | 学校服务器存储 | 个人钱包持有 |
| 验证方式 | 依赖学校系统查询 | 区块链公开验证，全球可查 |
| 可转让性 | 不可转让 | 个人NFT可转让 |
| 永久性 | 学校倒闭即丢失 | 永久保存，不依赖任何机构 |
| 稀缺性 | 可复制 | 独一无二，不可复制 |

**本质区别**：传统证书是学校"借给"你的，NFT是真正"属于"你的数字资产。

### Q2: 为什么选择ERC-1155而不是ERC-721？

**ERC-721的局限性：**
- 一个合约只能管理一种NFT（如只能管理毕业证书）
- 批量操作需要多次交易（发给10人需要10次交易，Gas费高）
- 适合CryptoPunks等单一类型收藏品

**ERC-1155的优势：**
- 一个合约管理多种NFT（官方4种 + 无限个人NFT）
- 支持批量转账（发给10人只需1次交易，省Gas）
- 节省部署成本（一个合约 vs 多个合约）

**本系统场景适配：**
- 需要管理：毕业证书、纪念徽章、荣誉证书、特殊奖项 + 无数个人纪念品
- 需要批量发放：一次给全班发毕业证书
- ERC-1155是天然的最佳选择

### Q3: 盲盒的随机数安全吗？可以被预测或操控吗？

**随机数生成来源：**
```
keccak256(
    block.timestamp +    // 当前区块时间戳
    block.prevrandao +   // PoS机制提供的随机数
    msg.sender +         // 调用者地址
    nonce                // 递增计数器
)
```

**安全性分析：**

| 因素 | 可控性 | 说明 |
|------|--------|------|
| block.timestamp | 矿工可微调（几秒误差）| 对盲盒结果影响极小 |
| block.prevrandao | 矿工无法预测 | PoS机制确保随机性 |
| msg.sender | 用户自己决定 | 固定值，不增加随机性 |
| nonce | 合约控制 | 确保每次调用不同 |

**结论**：对于盲盒场景足够安全。矿工无法操控prevrandao，用户无法预测未来区块的随机数。

**如果是涉及大额资金的场景**，建议使用Chainlink VRF（可验证随机函数），但对于毕业纪念盲盒，当前方案已足够。

### Q4: 保底机制具体怎么工作？请举例说明

**保底机制设计目的：**
- 防止运气极差（连续抽不到稀有）
- 心理安慰（知道不会永远非酋）
- 控制稀有NFT总量

**工作机制：**

```
每次铸造：pityCounter++（保底计数器+1）

if (pityCounter >= 20) {
    // 保底触发：第20次必出稀有或传说
    80%概率出稀有
    20%概率出传说
    pityCounter = 0（重置）
} else {
    // 正常概率
    70%普通 / 25%稀有 / 5%传说
    if (出稀有或传说) pityCounter = 0
}
```

**场景模拟：**

| 次数 | 结果 | pityCounter | 说明 |
|------|------|-------------|------|
| 1-19 | 连续19次普通 | 1→19 | 累积中... |
| 20 | 保底触发 | 0 | 必出稀有/传说 |
| 21 | 普通 | 1 | 重新累积 |
| 22 | 稀有 | 0 | 出稀有，重置 |
| 23-41 | 连续19次普通 | 1→19 | 再次累积... |
| 42 | 保底触发 | 0 | 再次保底 |

**保底的意义：**
- 最坏情况：连续19次普通，第20次必出好货
- 运气正常时：可能第3次就出传说，保底不触发
- 总量控制：传说约5-6%，稀有约25%，普通约70%

### Q5: 为什么官方NFT不能转让，个人NFT可以？

**官方NFT的本质**：学校对毕业生身份的官方认证

**如果可以转让的问题：**
- A把"毕业证书"转给B，B就成了冒牌毕业生
- 损害学校声誉和证书权威性
- 破坏身份认证体系的可信度

**个人NFT可以转让的原因：**
- 本质是纪念品，不是身份凭证
- 就像送一个毕业礼物，A可以送给B表达情谊
- 增加社交属性和趣味性

**类比理解：**
- 官方NFT = 身份证（不能转让）
- 个人NFT = 毕业纪念册（可以送人）

### Q6: 如何防止学生用脚本刷NFT？

**多层防护机制：**

1. **身份验证层**
   - 必须已注册为毕业生（需要管理员审核）
   - 一人一地址（地址唯一性校验）

2. **频率限制层**
   - 每日5次限制（链上强制执行）
   - 新的一天自动重置

3. **经济成本层**
   - 每次铸造需要Gas费（小额但有效）
   - 测试网免费，主网有实际成本

4. **随机数层**
   - 无法预测结果（不能定向刷传说）
   - 即使脚本也无法保证收益

**如果还是被刷怎么办？**
- 监控异常地址（铸造频率过高）
- 管理员可暂停铸造功能
- 分析链上数据，识别脚本特征
- 考虑增加验证码或人工审核

3. 解析事件数据，重新同步到数据库
4. 恢复完成（可能需要几小时，取决于数据量）

**设计原则**：区块链是"真相源"，数据库只是"缓存层"。即使数据库完全丢失，也可以从区块链完全重建。

### Q9: 每日5次限制是怎么实现的？为什么不是更多？

**实现逻辑：**
```solidity
// 链上存储
mapping(address => uint256) lastMintDay;    // 上次铸造日期
mapping(address => uint256) dailyMintCount; // 今日已铸造次数

// 铸造函数内
uint256 today = block.timestamp / 1 days;
if (today != lastMintDay[msg.sender]) {
    dailyMintCount[msg.sender] = 0;  // 新的一天，重置
    lastMintDay[msg.sender] = today;
}
require(dailyMintCount[msg.sender] < 5, "Daily limit reached");
dailyMintCount[msg.sender]++;
```

**为什么是5次？**
- **防止脚本刷取**：限制总量，控制NFT通胀
- **增加稀缺性**：不能无限铸造，才有收藏价值
- **鼓励日常参与**：每天来铸造，增加活跃度
- **平衡体验和经济**：5次足够体验，又不至于泛滥

**为什么是每日重置？**
- 自然时间周期，用户容易理解
- 北京时间0点重置（合约自动计算）
- 给用户"明天再来"的期待感
